Chapter 10: Testing and Debugging

System Error- Issue with system or external devices the program is interacting with

Programmer Error- Problem in the code. Logic or syntax issues

User Error- User enters bad data that can't be handled by the program

Stack trace displays the functions wrapping where the error took place

Warnings are issues that aren't severe enough to break the program but most of the time should be corrected

Strict mode is good because it keeps the code from using depricated/vunerable code and will error if there are normally non-breaking code, which avoids not seeing errors but having the results of them

linting can be used as an addon to many editors/IDEs or sometimes can be run as a command line opperation. Linting allows code style and formatting to be checked against rules. Very useful in multi-coder projects

Feature detection should be used to verify that dependancies are supported on the client's side before trying to load them.,

Try block of code -> Catch Exceptions -> Finally, run whatever code you need whether exception was thrown or not

Test-driven Development. Start by writting the tests the program needs to pass then write the code to pass those tests

Jest simplifies making Test-driven Dev not require so much extra work
